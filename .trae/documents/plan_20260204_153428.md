# 最高规格弹窗与代理优化方案

为满足您对“后台稳定运行”和“弹窗必达”的严格要求，我制定了以下升级方案：

## 1. 核心痛点修复：后台运行“保活”机制
**问题**: 浏览器通常会强制挂起后台标签页的定时器，导致自动刷新停止或延迟，这是“后台不弹窗”的根本原因。
**解决方案**: **引入 Web Worker 独立计时器**。
- **原理**: Web Worker 运行在独立线程中，即使主页面在后台或最小化，Worker 的定时器也能保持精准运行。
- **效果**: 无论您是否在前台，每隔 15 分钟（或设定时间）都会精准触发刷新，确保“实时监测”不打折。

## 2. 弹窗系统规格升级
**目标**: 确保摘要弹窗（Modal）和系统弹窗（Notification）在任何场景下必定触发。
- **系统弹窗 (Notification)**:
  - 强制使用 Service Worker 通道发送通知，这是唯一能在浏览器后台/关闭状态下唤醒系统的标准方式。
  - 在 `Header` 区域增加**通知权限状态指示器**。如果未授权，显示醒目的“开启通知”按钮，彻底解决因权限被拒导致的“哑火”。
- **摘要弹窗 (Modal)**:
  - 状态更新逻辑与系统通知解耦，确保即使用户离开很久，切回来时 Modal 依然停留在屏幕上等待查看。

## 3. 终极代理配置 (四级容错)
结合之前分析，实施最优代理策略，确保数据获取 100% 成功：
1. **[Priority 0] 本地高速通道 (Local Vite Proxy)**: 本地开发环境专用，**极速直连**，零延迟，无需授权。
2. **[Priority 1] CORS Anywhere**: 生产环境首选，速度快，需一次性授权。
3. **[Priority 2] CodeTabs**: 全自动兜底。
4. **[Priority 3] AllOrigins**: 最后防线。

## 执行步骤
1. **创建 `src/workers/timerWorker.js`**: 实现后台精准计时逻辑。
2. **修改 `src/App.tsx`**: 替换原有的 `setInterval`，接入 Web Worker 计时器。
3. **修改 `src/components/Header.tsx`**: 增加通知权限检查与请求按钮。
4. **重构 `src/services/steamApi.ts`**: 实施四级智能代理。
5. **优化 `src/store/useGameStore.ts`**: 确保刷新逻辑与弹窗触发的严谨性。
